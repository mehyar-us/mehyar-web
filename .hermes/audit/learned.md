## 2026-07-09 · turn-016 (sha 6dd002f)
Add-on SEO wins belong in `scripts/route-jsonld.json`, not in TSX components. The injection script is idempotent (marker `data-route-jsonld=<route>`), validated by the build, and survives no-JS crawlers — whereas runtime `SeoManager.tsx` only fires on hydration. Also: state.md drift — when the loop skips a tick-id bump (turn-015's rss feed landed without bumping state.md from 14), the next tick must reconcile. Always re-read state.md at the top of the tick and update the count even if "nothing changed semantically" — drift causes downstream confusion when the iteration diary and tick_id disagree.

CF Pages deploy lag observed today: ~5min 30s from `git push origin main` to `curl -sL https://mehyar.us/about/` showing the new bundle. Slightly longer than the 3-5min estimate in turn-014 — likely because /about isn't a hot path so CF may not warm the worker as eagerly. Two reinforcing lessons from this tick:

1. **Cross-route smoke after every additive SEO tick.** After injecting a new route-jsonld entry, hitting ALL 10 jsonld-equipped routes (not just the new one) catches any chance the idempotent marker logic failed or that the build broke a sibling shell. The loop in turn-022 ran the smoke in ~5s and confirmed every existing marker (1) still served, while /blog (intentionally no schema) still served marker=0. Pattern: `for r in / /about /micro-offer /booking /services /portfolio /contact /blog/<3 posts>; do curl -sSL https://mehyar.us${r} | grep -c "data-route-jsonld=\"${r}\""; done`. Add to the post-deploy checklist for any tick that touches scripts/route-jsonld.json.

2. **The `deploy:pages` npm script targets `gh-pages`, but the actual deploy target is Cloudflare Pages from `main`.** The script ran exit 0 with no output on turn-022 — it was running the wrong pipeline (its `git push -f origin HEAD:gh-pages` either got no output because git init in dist/ already had the remote, or silently failed at the prerequisite check). The real deploy is CF Pages auto-deploying from main on push. Don't trust `npm run deploy:pages` as the deploy mechanism; verify via `git ls-remote origin main` for the merge sha + curl-poll the live route for the new schema marker. The `deploy:pages` script is vestigial — leave it in place but never treat its silent exit-0 as proof of deploy.

Lesson: **preserve original file format when programmatically editing JSON config files.** `python -c "json.dump(...)"` re-sorts keys and re-indents, generating a 750+1013 line diff for a ~90-line semantic addition. Two specific gotchas: (1) the route-jsonld.json file uses CRLF line endings — `json.dump` defaults to LF, so the first naive pass silently changed line endings across all 1441 lines; (2) `json.dump` re-sorts dict keys, generating diff noise on every existing block. The right pattern is targeted regex splice for small additions, or re-serialize once and live with the bounded diff noise. For turn-024 I matched the CRLF + insertion-order preservation (inserted /blog after /about without touching the other keys' serialization). Diff still large because nested dict literals get re-flowed by json.dump, but semantically equivalent (provable: same 17 routes, same content, only /blog added).
